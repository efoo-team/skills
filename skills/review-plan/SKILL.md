# Review Plan

実装前の計画を、独立した複数のレビュー用サブエージェント(Oracle)で検証するためのスキル。
このスキルは read-only の計画レビュー専用であり、ソースコードの実装やワークツリー上のファイル編集は行わない。

## Instructions

1. レビュー対象の計画全体を読み、目的・前提・変更範囲・成功条件を把握する。
2. まず新規レビュー用サブエージェント(Oracle)セッションを1つ起動し、「この計画を厳密レビューするための重要観点」を洗い出してもらう。このサブエージェントは read-only とし、計画・設計文書・ソースコードの作成、編集、削除、実装を一切行わせない。
3. 計画が外部仕様、外部API、フレームワーク、ライブラリ、プロトコル、プラットフォーム制約に依存する場合は、調査できる範囲で外部仕様・公式ドキュメント・外部知見との適合性を確認するため、英語・中国語・日本語それぞれの外部仕様調査担当サブエージェントを別々の新規セッションで起動する。これは単なる翻訳比較ではなく、日本語プロジェクトの文脈、日本語圏の実務知見、英語・中国語圏に多い最新知見や深い外部知見を分離して収集し、単一言語の調査バイアスを下げるために行う。各セッションには対象言語を1つだけ割り当て、公式仕様・公式ドキュメント・主要リポジトリ・信頼できる外部知見を優先して調査させる。調査結果には、参照元の種別、URL、対象バージョンまたは公開日、該当セクション、normative / informative の区別、MUST / SHOULD / MAY などの要求レベル、調査できた範囲と未確認事項を含める。
4. 洗い出された観点と外部仕様調査結果から、独立して検証すべき観点を3つ以上選ぶ。以下は観点の例であり、必須チェックリストとして扱わない。レビュー用サブエージェント(Oracle)が挙げた計画固有のリスクや、ここにない重要観点を優先してよい。
   - 責務境界・モジュール境界が妥当か
   - 現在の問題だけに過度に最適化された局所実装になっていないか
   - 命名が目的・動詞・処理都合に寄りすぎず、概念や責務を表しているか
   - 計画が実行可能で、検証手段と完了条件が明確か
   - 外部仕様・公式ドキュメント・外部知見と計画の前提や実装方針が矛盾していないか
5. 観点ごとに別々の新規レビュー用サブエージェント(Oracle)セッションを起動し、フラットな前提で厳密レビューしてもらう。既存セッションは再利用しない。各レビュー用サブエージェントの責務は、問題点・根拠・リスク・確認事項の提示に限定し、計画の書き換え、ソースコード変更、実装作業、ファイル操作を禁止する。
6. レビュー用サブエージェント(Oracle)の指摘をいったん統合し、計画文書の改訂候補・未解決リスク・追加確認事項に分けて整理する。この時点では指摘を確定事項として扱わない。
7. 別の新規レビュー用サブエージェント(Oracle)セッションを起動し、統合した指摘の妥当性・優先度・対応方針を検証してもらう。誤検知、過剰設計、計画目的との不整合、対応コストに見合わない指摘を必ず疑う。このサブエージェントも read-only とし、採用判断や修正作業を直接行わせない。
8. 妥当性確認の結果を踏まえ、採用する指摘・保留する指摘・採用しない指摘を分け、最適な対応方針を決める。ただし、このスキルの実行中にソースコードの実装やワークツリー上のファイル編集を行ってはならない。採用すると決めた指摘は、そのサイクル内で必ず対応し、未対応のまま次サイクルへ進んではならない。
9. 採用すると決めた妥当性のある指摘への対応は、必要に応じてタスク分割し、タスク単位の別セッションのサブエージェントに対応を委譲して進める。タスク分割は、サブエージェントのコンテキスト逼迫による作業品質の劣化を防ぐために行う。
10. 2〜9のレビューサイクルを最低3回繰り返す。各サイクルは、レビュー、妥当性確認、採否決定、採用指摘への対応完了までを1単位とする。採用指摘が1件でも未対応の場合は、新しい観点のレビューへ進まず、そのサイクル内で対応を完了する。レビューサイクル中に実装担当サブエージェントを起動してはならない。
11. 最後に、実装前に直すべき点、実装中に監視すべき点、許容できる残留リスクを簡潔に提示する。必要な修正は提案として記述し、このスキルの実行中には適用しない。

## Rules

- レビュー用サブエージェント(Oracle)を呼び出すときは、観点ごとに必ず新しいセッションを使う。
- すべてのレビュー用サブエージェント(Oracle)、妥当性確認担当サブエージェント、観点洗い出し担当、外部仕様調査担当は read-only とする。ファイルの作成・編集・削除、ソースコードの変更、実装、コマンドの副作用を伴う操作を一切行ってはならない。
- レビュー用サブエージェント(Oracle)を起動するプロンプトには、必ず「DO NOT IMPLEMENT. DO NOT EDIT FILES. Review only. Return findings, evidence, risks, and questions as text.」を含める。
- 観点の異なるレビューを1つのレビュー用サブエージェント(Oracle)にまとめない。
- 外部仕様調査担当サブエージェントは、英語・中国語・日本語で必ず別々の新規セッションに分け、1つのセッションに複数言語の調査をまとめない。
- 外部仕様調査では、公式仕様・公式ドキュメント・主要リポジトリを優先し、言語別の外部知見は根拠の強さと確認範囲を明示する。
- 外部仕様調査では、公式仕様、公式ドキュメント、主要プロジェクト文書、互換性情報、コミュニティ記事を区別し、コミュニティ記事や非公式翻訳を仕様適合性の一次根拠にしない。
- 外部仕様調査では、対象バージョン、公開日、文書の成熟度、errata、廃止・代替・更新状況、地域差・言語差・ランタイム差を確認できる範囲で明示する。
- 外部仕様の MUST / MUST NOT / SHOULD / MAY、必須・推奨・任意、normative / informative を同じ強さで扱わない。
- 外部仕様との不適合を指摘する場合は、該当仕様・ドキュメント・外部知見の根拠を示し、調査できなかった場合は未確認事項として扱う。
- レビュー用サブエージェント(Oracle)の指摘を鵜呑みにせず、別のレビュー用サブエージェント(Oracle)による妥当性確認を必ず挟む。
- レビュー、妥当性確認、採用指摘への対応はそれぞれ別の責務として扱い、同じサブエージェントに兼任させない。
- 各レビューサイクルの終了条件は、発見された全指摘が採用・保留・不採用のいずれかに分類され、採用した指摘への対応がそのサイクル内で完了していることである。
- 採用する指摘がある場合のレビューサイクル終了条件は、実装やファイル編集の完了ではなく、採用理由・対応内容・受入基準・未確認事項を含むレビュー成果物が作成されていることである。
- 採用指摘を次サイクルへ持ち越してはならない。不要と判断した指摘は採用せず、不採用理由を記録する。
- 採用する指摘がない場合は、対応成果物を作成せず、親エージェントが次のレビューサイクルを開始してよい。
- レビュー担当サブエージェントは、計画修正や対応案の具体化を直接行わず、採用判断の材料となる指摘・根拠・リスク・確認事項の提示までに責務を限定する。
- 採用指摘が多い場合は、1つのサブエージェントにまとめて対応させず、対象領域・責務・ファイル範囲・受入基準ごとにタスクを分割し、タスク別のサブエージェントへ委譲する。これはコンテキスト逼迫による見落としや作業品質低下を避けるために必須とする。
- 採用指摘への対応で複数のサブエージェントを並列実行する場合は、各サブエージェントのプロンプトに「同時に他の作業者が別タスクを進めている。自分の担当範囲外の差分を削除・上書き・整形・巻き戻ししてはならない。他作業者の差分は意図しない差分として扱わず、触らずに残す。」という注意を必ず含める。
- このスキルの実行中に、レビュー担当・妥当性確認担当・観点洗い出し担当・外部仕様調査担当へ採用指摘の対応を兼任させてはならない。採用指摘への対応は、別責務のタスクとして、タスク単位の別セッションのサブエージェントに委譲する。
- 指摘は「どこが問題か」「なぜ問題か」「どう直すべきか」に分ける。
- 事実と推測を混ぜず、確認済みの根拠・未確定の仮説・追加確認が必要な点を分ける。
- 指摘の採否を決めた理由を残す。
- 問題がない場合も、確認した観点と問題なしと判断した根拠を書く。
