# Review Plan

実装前の計画を、独立した複数のレビュー用サブエージェント(Oracle)で検証するためのスキル。

## Instructions

1. レビュー対象の計画全体を読み、目的・前提・変更範囲・成功条件を把握する。
2. まず新規レビュー用サブエージェント(Oracle)セッションを1つ起動し、「この計画を厳密レビューするための重要観点」を洗い出してもらう。
3. 計画が外部仕様、外部API、フレームワーク、ライブラリ、プロトコル、プラットフォーム制約に依存する場合は、調査できる範囲で外部仕様・公式ドキュメント・外部知見との適合性を確認するため、英語・中国語・日本語それぞれの外部仕様調査担当サブエージェントを別々の新規セッションで起動する。これは単なる翻訳比較ではなく、日本語プロジェクトの文脈、日本語圏の実務知見、英語・中国語圏に多い最新知見や深い外部知見を分離して収集し、単一言語の調査バイアスを下げるために行う。各セッションには対象言語を1つだけ割り当て、公式仕様・公式ドキュメント・主要リポジトリ・信頼できる外部知見を優先して調査させる。調査結果には、参照元の種別、URL、対象バージョンまたは公開日、該当セクション、normative / informative の区別、MUST / SHOULD / MAY などの要求レベル、調査できた範囲と未確認事項を含める。
4. 洗い出された観点と外部仕様調査結果から、独立して検証すべき観点を3つ以上選ぶ。以下は観点の例であり、必須チェックリストとして扱わない。レビュー用サブエージェント(Oracle)が挙げた計画固有のリスクや、ここにない重要観点を優先してよい。
   - 責務境界・モジュール境界が妥当か
   - 現在の問題だけに過度に最適化された局所実装になっていないか
   - 命名が目的・動詞・処理都合に寄りすぎず、概念や責務を表しているか
   - 計画が実行可能で、検証手段と完了条件が明確か
   - 外部仕様・公式ドキュメント・外部知見と計画の前提や実装方針が矛盾していないか
5. 観点ごとに別々の新規レビュー用サブエージェント(Oracle)セッションを起動し、フラットな前提で厳密レビューしてもらう。既存セッションは再利用しない。
6. レビュー用サブエージェント(Oracle)の指摘をいったん統合し、修正候補・未解決リスク・追加確認事項に分けて整理する。この時点では指摘を確定事項として扱わない。
7. 別の新規レビュー用サブエージェント(Oracle)セッションを起動し、統合した指摘の妥当性・優先度・対応方針を検証してもらう。誤検知、過剰設計、計画目的との不整合、対応コストに見合わない指摘を必ず疑う。
8. 妥当性確認の結果を踏まえ、採用する指摘・保留する指摘・採用しない指摘を分け、最適な対応方針を決める。
9. 実際の計画修正や対応案の具体化は、別の実行担当サブエージェントに委譲する。レビュー担当サブエージェントと同じセッションに実対応をさせない。
10. 2〜9のレビューサイクルを最低3回繰り返す。各サイクルでは前回の統合結果と対応結果も入力し、見落とし・矛盾・過剰設計を再確認する。
11. 最後に、実装前に直すべき点、実装中に監視すべき点、許容できる残留リスクを簡潔に提示する。

## Rules

- レビュー用サブエージェント(Oracle)を呼び出すときは、観点ごとに必ず新しいセッションを使う。
- 観点の異なるレビューを1つのレビュー用サブエージェント(Oracle)にまとめない。
- 外部仕様調査担当サブエージェントは、英語・中国語・日本語で必ず別々の新規セッションに分け、1つのセッションに複数言語の調査をまとめない。
- 外部仕様調査では、公式仕様・公式ドキュメント・主要リポジトリを優先し、言語別の外部知見は根拠の強さと確認範囲を明示する。
- 外部仕様調査では、公式仕様、公式ドキュメント、主要プロジェクト文書、互換性情報、コミュニティ記事を区別し、コミュニティ記事や非公式翻訳を仕様適合性の一次根拠にしない。
- 外部仕様調査では、対象バージョン、公開日、文書の成熟度、errata、廃止・代替・更新状況、地域差・言語差・ランタイム差を確認できる範囲で明示する。
- 外部仕様の MUST / MUST NOT / SHOULD / MAY、必須・推奨・任意、normative / informative を同じ強さで扱わない。
- 外部仕様との不適合を指摘する場合は、該当仕様・ドキュメント・外部知見の根拠を示し、調査できなかった場合は未確認事項として扱う。
- レビュー用サブエージェント(Oracle)の指摘を鵜呑みにせず、別のレビュー用サブエージェント(Oracle)による妥当性確認を必ず挟む。
- レビュー、妥当性確認、実対応はそれぞれ別のサブエージェントに分ける。
- 指摘は「どこが問題か」「なぜ問題か」「どう直すべきか」に分ける。
- 事実と推測を混ぜず、確認済みの根拠・未確定の仮説・追加確認が必要な点を分ける。
- 指摘の採否を決めた理由を残す。
- 問題がない場合も、確認した観点と問題なしと判断した根拠を書く。
