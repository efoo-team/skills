## レポート出力フォーマット

### ヘッダ指標（冒頭に必ず記載）

```
事実ベース分析度: <0-100>/100
非忖度度: <0-100>/100
コメント: <一言コメント（最大一行）>
```

### 📊 レビュー件数サマリー

各AIレビュアーからの指摘件数を表示。0件のレビュアーは記載を省略。

```
Codex: N件 | CodeRabbit: N件 | Copilot: N件 | Claude: N件
```

### 📋 指摘一覧

指摘を上から順に列挙。1指摘 = 1エントリで完結。

**除外対象**: ドキュメント更新の報告、単なる通知、レビュー指摘ではない情報提供などはレポートから除外する。

**重大度アイコン**（IDの直後に必ず付与）:

- 🔴 Blocker / Major
- 🟡 Minor
- ⚪ 対応しない
- ✅ 対応済み
- 👀 対応中

**各エントリのフォーマット**:

```
**ID-01** 🔴 Major — 指摘の要旨（1行で完結）
- 指摘AI: Codex, CodeRabbit, Copilot
- `対象ファイル:行番号`（thread の場合）または `PR-level comment`（issue_comment の場合）
- 判断根拠（1〜2文）
- ステータス: ✅ done / 👀 in_progress / ⏳ pending / ⚪ skip
```

**重複時のルール**: 複数AIが同一指摘をした場合、指摘AIに全てのAI名を列挙する（例: Codex, CodeRabbit, Copilot）。1つのAIのみの場合は単独で記載。

### 📌 サマリ（指摘一覧の最後に記載）

```
- **対応済み（N件）**: ID-01, ID-02, ...
- **対応中（N件）**: ID-05, ...
- **対応しない — 誤指摘（N件）**: ID-09（理由）, ... [resolved]
- **保留（N件）**: ID-04（確認事項）, ...
- **未処理（N件）**: ID-10, ...（次回ループで処理）
```

## 対応方針の基本ルール

### 積極対応（Proactive Fix）

- デフォルトは「対応する」。些細な指摘であっても、将来の技術的負債を残さないことを最優先とする。
- 「対応しない」と判断するには、以下のいずれかの明確な根拠が必要:
  - 指摘内容が事実誤認である（根拠を明示）
  - 対応することで既存機能にリグレッションリスクが発生し、かつそのリスクが負債放置のリスクを上回る
- 指摘が妥当であれば、当初の変更意図に含まれていなかった論点であっても対応する。PRのスコープを理由に妥当な指摘を放置しない。
- 「保留」は、情報不足で判断不能な場合のみ使用する。判断可能なものは保留にせず対応する。
- nit・スタイル・命名などの軽微な指摘も「小さな負債の蓄積が将来の大きなコストになる」という前提で対応する。

### レビュアー特性の反映

| レビュアー                       | 特性                                              | 対応                                           |
| -------------------------------- | ------------------------------------------------- | ---------------------------------------------- |
| **codex**                        | 指摘数は少ないが重要度が高い                      | 優先的に確認、重要度高として扱う               |
| **coderabbitai bot**             | サマリー+コード行単位の詳細レビュー、指摘数が多い | 全指摘の妥当性を検証、事実と一致するものを採用 |
| **copilot**                      | コード品質・ベストプラクティス中心のレビュー      | 妥当性を検証し、有効な指摘を採用               |
| **github-actions bot（Claude）** | 誤りや誤解が混ざる可能性                          | 慎重に検証、妥当性が高いものは積極採用         |

### 妥当性の判断ルール

- すべてのレビュワー指摘について妥当性を検討し、妥当性が高いものは積極的に採用して改善方針に反映
- 判断が曖昧、または情報不足の場合は**ユーザーへ確認**してから方針を確定する
- `.md` への指摘は実装と比較し、ドキュメント更新不足の可能性を明示

## 客観性・非忖度ガイドライン

- **事実ベース**: 収集したPRコメントの一次情報（本文、種類、時刻、レビュアー、スレッド状態）に依拠し、推測は明確に分離する。
- **非忖度**: ユーザーの意図推測や口調に引きずられず、コメントの内容と根拠のみに基づいて判断する。
- **反証可能性**: 結論は引用可能な事実と紐づけ、代替仮説がある場合は最小限言及する。
- **仮定の明示**: 足りない情報に基づく推測は「仮定」と明言し、確度は低く評価する。
- **フェーズ分離の徹底**: 分析フェーズ（Phase 3-5）ではファイル変更・コミットをしない。対応はPhase 6で実施する。

## 評価指標（出力必須・0-100）

- **事実ベース分析度**: 事実への依拠度と推測の抑制度。一次情報の網羅性、矛盾点の指摘、仮定の明示によって評価。
- **非忖度度**: ユーザー発言への迎合を避け、コメント本文と実装事実に基づいて自立的に判断できた度合い。

スコアリング目安:

- 90-100: 重要点の抽出が事実に完全整合。推測は最小で明示。判断はユーザー意図から独立。
- 70-89: 大筋は事実整合だが一部推測あり。非忖度もおおむね維持。
- 40-69: 事実/推測の混在が目立つ、もしくは忖度の影響あり。
- 0-39: 根拠不十分、推測過多、忖度的判断が支配的。

## 禁止事項

以下の操作は**絶対に行ってはならない**:

1. **ノイズエントリへの操作禁止**: Phase 3 で「ノイズ」と判定した `issue_comment` に対して、Reaction 付与・コメント投稿・`gh pr-review-check resolve` の呼び出しを行ってはならない
2. **新規トップレベルコメントの投稿禁止**: PR に対する新規の `issue_comment`（トップレベルコメント）を投稿してはならない。コメントは必ず既存スレッドへの返信（`addPullRequestReviewThreadReply`）として行う
3. **resolve 不可能な type への resolve 禁止**: `type: issue_comment` および `type: review` に対して `resolveReviewThread` を呼び出してはならない（GitHub API の制約上エラーになる）
4. **人間レビュアー起票スレッドの自動 resolve 禁止**: 人間（非 bot）が起票したスレッドを自動的に resolve してはならない。返信と reaction のみ行い、resolve は人間レビュアーに委ねる
5. **push 失敗時の resolve 禁止**: テストやプッシュが失敗した状態でスレッドを resolve してはならない。`in_progress` のまま維持する
6. **重複操作の禁止**: 既に返信済み・reaction 済み・resolved 済みのエントリに対して、同じ操作を繰り返してはならない（冪等性チェック必須）

## 失敗時のハンドリング

- `gh` 未導入/未ログイン: `gh --version` と `gh auth login` の案内のみ（処理継続不可）
- `gh pr-review-check` 未インストール: インストールコマンドを案内
  ```bash
  gh extension install abekdwight/gh-pr-review-check
  ```
- PR特定不能: 引数の確認を依頼し、中断
- Analyzer 失敗: 部分結果で継続するかユーザーに確認

## 承認モード（オプション）

デフォルトでは承認不要で自律実行する（Phase 0 → 8 を連続実行）。

ユーザーが引数または指示で「承認してから対応して」「確認してから進めて」等と**明示的に指定した場合のみ**、以下の挙動に切り替わる:

1. Phase 3（レポート生成）まで実行し、レポートをユーザーに提示
2. ユーザーからの承認を待つ
3. 承認後、Phase 4 → 8 を実行
