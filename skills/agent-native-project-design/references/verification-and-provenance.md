## 1. 検証ゲート強度の4段階

検証の「強制力」は同じではない。重要度と失敗コストに応じて、以下の4段階から選ぶ。

| 段階 | 機構 | 強制力 | 適用場面 |
|---|---|---|---|
| 1 | **プロンプト内反復**：同一プロンプトで「実装後にテストを走らせ、通るまで直せ」と指示 | 助言（読み落とし得る） | 日常タスク。失敗しても再実行が安いもの |
| 2 | **goal 条件**：完了条件として宣言し、ターンごとに評価器が再チェック | 助言＋反復評価 | 長いタスクで「途中で目的を見失う」ことへの対策 |
| 3 | **Stop hook**：ターン終了時にスクリプトがチェックを実行し、失敗なら完了を決定的にブロック | 保証 | 「これを通さずに完了してはならない」もの（ビルド・スキーマ検証・lint） |
| 4 | **検証サブエージェント**：別コンテキストの採点者が結果を反証しにいく（作業者と採点者の分離） | 保証＋バイアス分離 | 作業者自身のコンテキストが実装にバイアスされている場合。無人実行が長いほど価値が上がる |

段階4の設計上の注意（出典: Claude Code 公式 best practices）:

- 検証サブエージェントには **diff と判定基準だけ**を渡し、変更を生んだ推論は見せない（fresh context）。
- ツールを Read / Grep / Glob 等の read-only に絞れば、「レビューアーが変更を加えられない」ことが構造的に保証される。
- 「gap を探せ」と指示されたレビューアーは健全な作業にも必ず何か報告する。**「正しさと明示された要件に影響する gap だけを flag し、残りは optional」**とスコープ制約を掛けないと、全対応が over-engineering（不要な抽象層・防御的コード・起こり得ないケースのテスト）を生む。

段階1でも、stdout ナッジで強度を底上げできる。`video-production-cooking-lesson/scripts/grab_frames.sh`（50-51行）はフレーム保存パス一覧の直後に「上記の画像を目視で確認してください。」を印字し、検証ルールをツール実行の瞬間にコンテキスト直近へ再注入する。

## 2. 証拠主義 — ツール実行と証拠観察の分離

エージェントは「ツールを呼んだ事実」を「検証済み」と錯覚しやすい。両リポジトリはこれを名指しで禁止する。

- cooking: 「切り出した画像は必ず自分で開いて目視してから判断する」（`.claude/skills/recipe-draft/SKILL.md` 60-61行）。「フレームを実際に切り出して目視していない情報に〔映像確認〕を付けることは、どんな場合もしてはいけない」（同 138-139行）。
- tabi: 「見ていない画を指定しない。未確認のものは『確認状態』欄で正直に申告する」（`docs/claude_ai_prompts_v2.2.md` §3）。

設計指針：成功の主張ではなく、**確認した証拠そのもの**（時刻・出力・URL・スクリーンショット）を成果物に要求する。証拠が値として残るマーカー（次節）は、この規律を成果物フォーマットの側から強制する仕掛けである。

## 3. provenance マーカーの設計（実例2件）

### 3.1 cooking：2用途予約の角括弧マーカー

`video-production-cooking-lesson/.claude/skills/recipe-draft/SKILL.md`（131-141行）は、成果物のすべての不確かな箇所に2種類のマーカーを義務付ける。

| マーカー | 意味 | 値として持つ証拠 |
|---|---|---|
| `〔映像確認：hh:mm:ss〕` | フレームを実際に目視して裏取りした情報 | 目視したフレームの時刻 |
| `〔要確認：理由〕` | 音声にも映像にも根拠がなく、人の確認が必要な情報 | なぜ確定できないかの理由 |

出力例（同 SKILL.md 例5・例6、255-265行）:

```
• 太白ごま油を目安18g計量する。〔要確認：発話は「18」のみ。値と単位を確認すること〕
• ナスを切り口を下にして、せいろに隙間なく並べる。〔映像確認：00:08:42〕
```

設計上の要点:

- **記法の予約**：角括弧〔〕はこの2用途のみに予約し、他のラベルは【】を使う（同 140-141行）。下流の機械的・目視的走査を壊さないため。
- **全主張の分類**：出力の各主張が「SRT由来／目視済み映像由来／未確定」のいずれかに必ず分類されるため、ハルシネーションが成果物の**構造レベル**で検出可能になる。
- **人間へのハンドオフプロトコルを兼ねる**：`README.md`（61-62行）は「後工程はこの2つを目印に仕上げてください」と人間側の手順を定める。時刻付き〔映像確認〕は Premiere のタイムラインでそのまま検証できる（時刻のズレ換算が不要な設計。README 59-60行）。

### 3.2 tabi：出所分類語彙＋「確認状態」必須カラム

`video-production-tabi` は証拠の種別（フレーム/シートで**視認した事実** vs **SRT由来**の発言）を分類し、確信度の語彙を定義する（`CLAUDE.md`「検証ルール（絶対）」、`docs/claude_ai_prompts_v2.2.md` §3）:

- `【要確認】`／「推定」／「判読不能」／「裏取り済み（URL）」
- カットリスト（`.claude/skills/cutlist/SKILL.md`）には**「確認状態」カラムが必須**：`フレーム確認済み／SRTのみ／要人間確認`
- 「裏取り済み（URL）」は公的に検証可能な事実（自治体の認定・生産量日本一・組織の正式名称等）に限定し、個人名・私人の情報は検索しないという調査行動の倫理境界も検証ルール内に埋め込まれている。
- フェーズによる強度調整：quick-draft（叩き台）では「固有名詞の裏取りは不要（【要確認】のまま使う）」、triage / cutlist（公開前提）では裏取りと確定表記を要求する。厳格さを常時最大にしない。

### 3.3 マーカー設計の一般則

1. **語彙を閉じる**（予約記法・enum 化）。自由記述にすると走査できなくなる。
2. **出所クラスは相互排他かつ全主張をカバー**する。「分類なし」を許すと未検証の主張がすり抜ける。
3. **検証行為の証拠（時刻・URL）をマーカーの値として持たせる**。人間が再検証できる形にする。
4. **人間の次アクションが読める**ようにする（理由・確認方法・件数サマリ）。マーカー設計＝ハンドオフプロトコル設計である。

## 4. 検証予算の数値化

上限のない検証指示は過剰検証ループを生む。cooking の実例（`SKILL.md` 40-48, 60-65行）は「発動条件・上限・許容誤差・フォールバック」の4点セットで予算を数値化している:

| 要素 | 規定 |
|---|---|
| 発動条件 | 映像確認は「音声だけでは〔要確認〕になる箇所」限定。用途を4つに列挙して閉じる（指示語の解決／誤変換の裏取り／画面上の文字・数値の読み取り／手順写真候補は今回不要）。「全編を漫然と見ない」 |
| 1論点あたり上限 | フレーム最大3枚。動きが論点なら 0.5〜1秒間隔で2〜3枚を見比べる |
| 許容誤差 | 低fpsプロキシの時刻誤差 ±0.1秒程度は「この作業では問題にならない」と明示 |
| 超過時フォールバック | 3枚見ても確定できなければ断定せず〔要確認〕を残す（検証の失敗が成果物の正規の状態になる） |

tabi でも同型の予算がある（`CLAUDE.md`、`docs/claude_ai_prompts_v2.2.md` §3）: 切り出しは「1カットにつき1〜3枚」、「無駄な切り出しでコンテキストを消費しないこと」、さらに**実施したフレーム切り出しを出力末尾に一覧で報告**させる（使用量の監査）。予算には消費の可視化まで含めるとよい。

## 5. 前提スモークテストと入力の信頼順位

### 5.1 同期スモークテスト（cooking）

パイプライン全体が「SRT のタイムコード＝動画の時刻」という前提に依存する。この前提が崩れると全成果物が無効になるため、本作業前に安価に検証する（`SKILL.md` 73-76, 199-205行）:

1. 入力健全性：SRT の最終時刻と動画の尺が大きく矛盾しないか確認する。
2. 同期確認：**序盤と終盤の発話2箇所**についてフレームと発話内容の整合を確認する。
3. **ズレ時の縮退動作**：出力冒頭にズレの注記を入れ、〔映像確認〕の採用を慎重にする（作業を止めるのではなく、信頼度を下げて続行する）。

前提検証は「2〜3点のサンプリング」で足りる。全数検証を前提に置くとスモークテスト自体が高コスト化する。

### 5.2 入力の既知欠陥分布と信頼順位（cooking）

入力（Premiere 自動生成 SRT）の**誤り方の傾向を実例付きで列挙**する（`SKILL.md` 16-25行）:「芯」→「心」「シーン」、「せいろ」→「正論」「世論」「制度」、「蒸す」→「無視」、相づち・雑談・話者混在。誤変換の実例を先に与えることで、エージェントはドメイン文脈による補正（「せいろ蒸しの動画なら『正論』は『せいろ』」）を事前分布として持てる。

複数情報源が矛盾したときの**信頼順位を全順序で明文化**する（同 190-195行）:

```
参考資料 ＞ 映像の読み取り ＞ 音声（SRT）
```

全順序があるため、矛盾に遭遇したときの判断が追加の指示なしで決まる。

## 6. 2パス構成 — 高コスト検証をターゲットに限定する

cooking の recipe-draft ワークフロー（`SKILL.md` 197-221行）は、高コストな検証行動（フレーム切り出し＋目視）を次の構造で限定する:

1. **パス1（計画・テキストのみ）**：SRT のみから叩き台を作りつつ、「映像で解決すべき箇所リスト」（指示語・誤変換の裏取り・画面上の数値）を作成する。
2. **パス2（検証・ターゲットのみ）**：リストの各箇所だけをフレーム照合する。解決できた箇所は本文に反映して〔映像確認〕を付け、できなかった箇所は〔要確認〕を確定する。
3. **最終通読**：全体を通読して、手順の順番・粒度・印の付け漏れを点検する（自己レビューパス）。

tabi は同じ発想を「検証コストを主張のリスクに比例させる」形で実装する（`.claude/skills/cutlist/SKILL.md`）: 「この画を使う」と指定する箇所（下流の編集作業に直接影響する主張）だけ原則フレームを切り出して実物確認し、会話中心のカット（SRT で内容が確定するもの）は SRT のみで可 — ただし「確認状態: SRTのみ」と申告させる。

## 7. 抜き取り検査による人間レビュー

人間レビューを全数検査として設計すると、人間が検証ボトルネックに戻る。tabi は**サンプリング合否判定**として設計する（`.claude/skills/triage/SKILL.md`「終わり方」、`.claude/skills/cutlist/SKILL.md`）:

- 台帳・カットリストの**時刻を2〜3点だけ** Premiere で実際に当てて「記述と画が合っているか」を確認する。これはエージェントの主張全体の信頼性を推定する合否判定（「見えないものを見たと書いていないか」の検査）。
- **エージェント自身にこの検査をユーザーへ勧めさせる**（「台帳の時刻2〜3点をPremiereで実際に当てて…確認することをユーザーに勧める」）。レビュー導線を成果物の「終わり方」に組み込むことで、検査の実施が人間の記憶力に依存しなくなる。

抜き取りが成立する前提は、第3節の provenance マーカーである。全主張に出所と時刻が付いているから、任意の2〜3点を選んで突合できる。

## 8. バイアスへのカウンターウェイト

モデルには既知の系統的バイアスがあり、観察されたら**逆方向の力を明文化**して打ち消す。tabi の実例（`.claude/skills/triage/SKILL.md`、`.claude/skills/quick-draft/SKILL.md`、`docs/claude_ai_prompts_v2.2.md` §5）:

| バイアス | カウンターウェイト指示 |
|---|---|
| 全肯定（採用度が◎○に偏る） | 「◎○に偏らせず、×と△を必ず検討する。**迷ったら△**」— デフォルトまで指定して判断の逃げ道を塞ぐ |
| 削除忌避 | 「『削る』を恐れない。5時間素材は過半を削る前提で判定する」 |
| レビューアーの過剰指摘（gap を探せと言われると健全な作業にも必ず指摘する） | 「正しさと明示された要件に影響する gap だけを flag し、残りは optional として扱え」（Claude Code 公式） |

これらは机上の想定ではなく、運用で観察された偏りへの対策として追加された（tabi の変更履歴 §5「採用度が◎○に偏らないよう『×と△を必ず検討』を明記」）。第13節のインシデント駆動運用の一部である。

## 9. 発散 → ハードストップ → 収束

人間ゲートは「高価な工程の直前」に1点だけ置く。tabi の cutlist ワークフロー（`.claude/skills/cutlist/SKILL.md` STEP 2）:

1. **発散（安価）**：構成3案を提示する。3案には「切り口を変えること（人物軸／食材軸／土地の物語軸）」という多様性の強制が付く。
2. **ハードストップ**：「**★ここで必ず停止し、ユーザーの選択・修正指示を待つ。**」— 停止は太字・記号付きで指示され、例外がない。
3. **収束（高価）**：選ばれた案のみをタイムライン化・カットリスト化する。

補完パターンとして、quick-draft は「希望確認（最初に1回だけ）」で確認質問を冒頭に集約し、以降は自律的に走らせる（確認回数の予算化）。確認質問は「制作者にしか答えられないものに絞り、選択肢つきで」出させる（`.claude/skills/triage/SKILL.md`）。途中で散発的に質問が飛ぶ設計は、人間の割り込みコストと待ち時間の両方を増やす。

## 10. 失敗ケースを含む few-shot

成功例だけの few-shot は「必ず解決してしまう」バイアスを植え、エージェントが確定できない箇所を推測で埋める誘因になる。

cooking の `SKILL.md` 末尾の例1〜例7（222-271行）は本文ルールとほぼ1対1対応しており、**例7が「映像でも確定できない場合は〔要確認〕を維持する」という失敗時の模範解答**である（267-271行）。「検証を試みたが解決しなかった」ときの正しい出力を実例で示すことで、〔要確認〕を残すことが失敗ではなく正規の完了経路であることを学習させる。

few-shot を設計するときは、少なくとも1例を「未解決で終わる例」にすること。

## 11. 黙った歪曲の禁止とエスカレーション規則

制約に収まらないとき、エージェントは黙って品質を犠牲にして制約に適合しがちである。エスカレーションの条件と形式を数値で規定する（tabi の実例）:

- **尺の±20%ルール**：「尺が目標を±20%以上外れる場合は、無理に収めず『削り候補』を先に提示してユーザーの判断を仰ぐ」（`.claude/skills/cutlist/SKILL.md`）。超過の事実＋判断材料（削減候補）をセットで出させる。
- **前提欠如時の停止**：SRT が無ければ推測で進まず、「Premiereからの書き出し（同一シーケンス・同一範囲）を依頼して止まる」（`.claude/skills/quick-draft/SKILL.md` 手順1）。必要物の依頼まで含めて停止する。
- **意味の追加の禁止**：発言テロップは「語尾の整文は可、意味の追加は不可」— 生成モデルの「気を利かせた補完」を成果物仕様で禁止する。
- cooking 側の同型ルール：「発話にも映像にも根拠のない情報を、推測で書き加えない。特に、誤った調理アドバイスを『それらしく』作り出してはいけない」（`video-production-cooking-lesson/CLAUDE.md` 検証ルール）。**ドメイン最大のリスクを名指しで禁止**している点が重要。

## 12. モダリティ限界の明文化

エージェントが使う知覚手段の限界そのものをルールにする。ツールが返すものと、そこから正当に主張できることの境界を指示に含める。

- **静止画から動きを断定しない**：「静止画から連続した動き（混ぜる方向・速さ・力加減）を断定しない。フレームに見えている事実だけを採用する」（cooking `CLAUDE.md` 検証ルール、`SKILL.md` 69-70行）。静止画3枚から動きは「もっともらしく推測できてしまう」ため、能力の過信をルールで抑制する。同時に限界内の代替手段（動きが論点なら0.5〜1秒間隔で2〜3枚を見比べる）も与える — 禁止と代替のペア。
- **サンプルの補間を禁止する**：「コマは（一定間隔の）サンプルにすぎません。コマとコマの間に何があるかを断定しないでください」（tabi `docs/claude_ai_prompts_v2.2.md` §2、`CLAUDE.md` にも同旨）。証拠がサンプリングであるなら、その説明とセットで補間禁止を与える。
- **聴けない・読めないものを列挙する**：「動画の音声は聞けない前提（発話情報はSRTがすべて）」（cooking）。tabi は知覚限界と代替手段を1対1で対応付ける（動画再生不可→フレーム切り出し、音声不可→SRT＋RMS表）。さらに「RMSは0に近いほど大きい音」のような**読み方の較正**をデータファイル自体の先頭コメントにも書き込む（データの自己記述化）。

## 13. インシデント駆動の変更履歴

検証ルールは机上で網羅するものではなく、実際に起きた失敗から漸進的に増やす資産である。tabi の `docs/claude_ai_prompts_v2.2.md` §5・§7 は、**各規則に「なぜできたか」（事故・観察）を紐づけた変更履歴**を残している:

| 起きた失敗 | 還元されたルール |
|---|---|
| 初回テストで part 番号の取り違え（ハルシネーション）が実際に発生 | manifest.txt を唯一の正とし、part 番号への言及には必ず元動画の時間区間（HH:MM–HH:MM）を併記させる |
| 採用度評価が◎○に偏る | 「×と△を必ず検討。迷ったら△」を明記 |
| 見ていない画をカットリストに指定する | 「確認状態」カラムの必須化＋フレーム切り出しの規律 |
| ツールが drawtext 非搭載を「フォント未検出」と誤診断 | 能力検出とエラーメッセージの修正（`tools/split_for_claude_v2.py` docstring の修正メモに理由を記録） |

「なぜ」を残す効果は2つある。(1) 後続のプロンプト改訂者が、ルールを恣意的な命令ではなく失敗からの学習として理解でき、安易な削除や形骸化を防げる。(2) モデルが賢くなったとき、「この失敗はもう起きないか」を再検証してルールを剪定する判断材料になる。ルールは失敗の証拠とともに増え、モデル更新とともに剪定される。
