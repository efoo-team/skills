# issue-report

不具合報告や機能要望をヒアリングで具体化し、GitHub issue として登録するスキル。

## 目的

曖昧なままの issue（どの画面のことか分からない・再現手順がない・期待と実際の区別がない・目的が分からない要望）が登録されると、開発者が報告者に聞き直す手戻りが発生する。本スキルは、報告者から平易な言葉で課題を聞き取り、コードベースの調査で対象を特定・確認し、「開発者がそのまま動ける品質」の issue を登録することを目的とする。

## 想定ユーザー

このスキルを起動するのは非エンジニアである前提で振る舞う。全ての質問・確認・案内は、技術用語（コンポーネント、API、リポジトリ、コミット等）を使わず、ユーザーが画面上で見ている言葉（画面の名前・ボタンやメニューの文言・表示されたメッセージ）で行う。

## 追加指示の扱い

ユーザーが引数として渡した内容（Claude では `/issue-report` の引数、Codex では `$issue-report` の引数）は、問題や要望の最初の説明として扱い、本スキル内の他の方針より優先して適用する。ただし「承認なしに issue を登録しない」「解決策・実装仕様を決めない」などの重要な制約に反する場合は、実行せず直ちに停止して確認を求める。引数が空の場合は、「どんな問題や、こうしてほしいという要望がありますか？」と自由に話してもらうところから始める。

## 基本方針

- **課題に徹する**: ヒアリングは「何に困っているか・何が起きたか・何のためか」の具体化に集中する。どう直すか・どんな仕様にするかは開発者が検討する領域であり、質問もしないし issue にも書かない（開発者が仕様を検討する余白を残す）
- **全ての質問はスキップできる**: 報告者が全てに答えられるとは限らない。質問には必ず「わからない」という回答を許容し、わからなかった項目は issue に「不明」と正直に書く。推測で埋めない
- **まず調査、それから質問**: 質問の前にコードベースを読み取り専用で調査し、対象の画面・機能の候補を特定してから確認する
- **確認なしに登録しない**: GitHub への書き込み（issue 登録）は、完成した本文をユーザーに見せて明確な承認を得てからのみ行う

## 実行手順

以下のチェックリストを応答に貼って進捗を追跡する:

```
- [ ] 1. 受け取りと初期調査（コードベースを読み取り専用で調査）
- [ ] 2. 対象の特定確認（課題の意味と場所の認識合わせ）
- [ ] 3. 課題のヒアリング（スキップ可能な質問）
- [ ] 4. issue 本文の作成（テンプレート優先）
- [ ] 5. 確認と登録（承認 → gh issue create → URL 報告）
```

### 1. 受け取りと初期調査

ユーザーの説明からキーワード（画面に表示されている文言・機能の呼び名・エラーメッセージ等）を抽出し、現在のリポジトリを読み取り専用で調査して対象候補を特定する。

- ユーザーの言葉は画面の言葉であるため、UI に表示される文言（ボタンラベル・見出し・メッセージ）でのテキスト検索が最も有効
- 画面・ルーティング・主要機能のディレクトリ構造も確認する
- serena 等のセマンティック検索が利用可能なら併用する

### 2. 対象の特定確認

調査結果をもとに、対象候補を最大5件、画面の言葉で提示して確認する。この段階ではファイルパス・技術用語をユーザーに見せない。

```markdown
## 🔍 おっしゃっている場所の確認

「[要望の要約]」について、関係しそうな画面・機能を調べました：

| No | 画面・機能 | 説明 |
|----|----------|------|
| 1 | [画面名・機能名] | [ユーザーから見た1行説明] |
| 2 | [画面名・機能名] | [ユーザーから見た1行説明] |

この中に、おっしゃっているものはありますか？

回答例:
- No.1 の「◯◯画面」のことです
- どれか分かりません（→ 操作の流れを一緒にたどって特定します）
- この中にはありません
```

候補が見つからない・ユーザーが特定できない場合は、「その画面を開くまでに、どのメニューやボタンを押しますか？」のように操作の流れを一歩ずつ聞いて絞り込む。それでも特定できない場合は「対象不明」のまま手順3に進んでよい（開発者が後から特定できるだけの情報をヒアリングで集める）。

**Gate**: 課題の意味と場所の認識（「おっしゃっている課題は◯◯という意味で、場所は◯◯で合っていますか？」）をユーザーと合わせるまで手順3に進まない。ただし「特定できなかった」という合意でもよい。

### 3. 課題のヒアリング

報告の種類（不具合系 / 要望系）で質問セットを分ける。質問は利用可能な質問機能で一度に最大4問、回答例つきで行い、**どの質問にも「わからない」を選択肢として含める**。「わからない」と答えられた項目はそれ以上追わない。

**不具合系（何かがおかしい・動かない）:**

| 聞くこと | 質問の例 |
|---|---|
| 実際に起きたこと | 何が起きましたか？（画面に出たメッセージがあれば、そのままの文で教えてください） |
| 期待していたこと | 本当ならどうなるはずだと思っていましたか？ |
| 再現手順 | どんな操作をしたら起きましたか？（順番に。途中までしか覚えていなくても大丈夫です） |
| 環境・頻度 | いつ頃から起きていますか？ 毎回起きますか？ スマホとパソコンのどちらで使っていましたか？ |
| 影響 | この問題で、いま何ができなくて困っていますか？ |

「期待していたこと」と「実際に起きたこと」は必ず分けて記録する（混ざっていると、開発者が仕様なのか不具合なのか判断できない）。

**要望系（こうしてほしい・機能が欲しい）:**

| 聞くこと | 質問の例 |
|---|---|
| 困りごと | いま、何に困っていますか？ どこが不便ですか？ |
| 目的 | それができると、何が良くなりますか？（例: 作業時間が減る、ミスが減る、お客様に説明しやすくなる） |
| 現在の回避策 | いまはどうやって対処していますか？ |
| 頻度・影響範囲 | どのくらいの頻度で困りますか？ 同じことで困っている人は他にもいますか？ |

**要望系で聞かないこと**: どんな画面・ボタン・仕様にしてほしいかという実現方法。仕様の検討は開発者の工程であり、非エンジニアに仕様の判断を委ねることはかえってリスクになる。目的がある程度抽象的でも、困りごとと目的が言語化できていれば十分とする。ユーザーが自発的に実現案を話した場合は「報告者のアイデア」として記録するだけにとどめ、深掘りしない。

### 4. issue 本文の作成

1. 対象リポジトリの `.github/ISSUE_TEMPLATE/` を確認し、該当する種類（不具合 / 要望）のテンプレートがあればその構成に従う
2. テンプレートが無ければ、後述の「標準 issue 形式」で作成する
3. タイトルは「どこで・何が」が一目で分かる形にする（例: 「注文履歴画面で注文一覧が表示されない」「レシピ検索の絞り込み条件を保存したい」）
4. 調査で特定した関連ファイル・機能は、本文末尾の「開発者向け参考情報」に書く（提示時に、報告者へ「この部分は開発者向けの技術メモです」と伝える）
5. わからなかった項目は削除せず「不明」と書いて残す（開発者が聞き直すべき点が見えるようにするため）

### 5. 確認と登録

1. 完成した issue のタイトルと本文をユーザーに見せ、「この内容で登録してよいですか？」と確認する。修正の要望があれば反映して再確認する
2. 承認を得たら、`gh issue create` で現在のリポジトリに登録する。ラベルは `gh label list` で既存ラベルを確認し、bug / enhancement 相当のものがあれば付与する（無ければラベルなしで登録する）
3. 登録した issue の URL をユーザーに伝える
4. **gh コマンドが使えない・認証されていない場合**: エラーで止めず、issue のタイトルと本文をコピーできる形で提示し、「GitHub でこのプロジェクトのページを開く → 上部の Issues タブ → 緑色の New issue ボタン → 貼り付けて Submit」という手動登録の手順を平易な言葉で案内する

## 標準 issue 形式

リポジトリに issue テンプレートが無い場合に使う。

**不具合用:**

```markdown
## 概要
[1〜2行で問題の要約]

## 対象の画面・機能
[画面名・機能名。特定できなかった場合は「不明（再現手順を参照）」]

## 実際に起きたこと
[起きたこと。画面に表示されたメッセージがあれば原文のまま]

## 期待していた動作
[本来どうなるはずだったか]

## 再現手順
1. [操作を順番に]
（不明な場合は「不明」と記載）

## 環境・頻度
- 使用環境: [スマホ / パソコン / ブラウザ名 など。不明なら「不明」]
- 発生頻度: [毎回 / ときどき / 一度だけ。不明なら「不明」]
- 発生時期: [いつ頃から]

## 影響
[この問題で誰が何をできなくて困っているか]

## 開発者向け参考情報
[調査で特定した関連ファイル・機能・処理など。特定できなければこのセクションごと省略]
```

**要望用**（実現方法・仕様のセクションを意図的に置かない。仕様は開発者が検討する）:

```markdown
## 困りごと・背景
[いま何に困っているか・なぜ必要か]

## 目的
[実現すると何が良くなるか]

## 対象の画面・機能
[関連する既存の画面・機能。既存機能と関係しない場合は「新規」]

## 現在の回避策
[いまどう対処しているか。無ければ「なし」]

## 頻度・影響範囲
[どのくらいの頻度で困るか・誰が困っているか]

## 報告者のアイデア（参考）
[報告者が自発的に述べた実現案があれば記載。仕様の決定ではなく参考情報。無ければこのセクションごと省略]

## 開発者向け参考情報
[調査で分かった関連機能・類似の既存実装など。あれば記載、無ければこのセクションごと省略]
```

## 重要な制約

- ユーザーの明確な承認を得るまで issue を登録しない。GitHub への書き込みは `gh issue create` のみとし、それ以外の書き込み操作（issue の編集・クローズ・コメント等）は行わない
- ソースコード・設定ファイルの編集・作成・削除は行わない（調査は読み取り専用）
- 解決策・実装方法・仕様案を自分から提案しない・issue に書かない（報告者が自発的に述べた案のみ「報告者のアイデア」として区別して記録する）
- 質問への回答を強制しない。わからない項目は「不明」と明記する
- ユーザーへの質問・確認に技術用語・ファイルパスを使わない（issue 本文の「開発者向け参考情報」セクションのみ例外）
- ユーザーの発言を要約・言い換えしたときは、必ず「この理解で合っていますか？」と確認してから issue に書く。曖昧な報告を確認なしに具体的な内容へ勝手に翻訳しない
