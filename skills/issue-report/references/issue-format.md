# issue-report: issue 本文形式と書き換え手順

起草ワーカー（縮退時は parent）が Phase 6 で従う本文形式と、統合（既存 issue の本文書き換え）の手順。

## 共通規約

- 対象リポジトリの `.github/ISSUE_TEMPLATE/` を確認し、該当する種類（不具合 / 要望）のテンプレートがあればその構成を優先する。無ければ本ドキュメントの標準形式を使う。テンプレートを使う場合も「対象領域の定型形式」と「更新履歴」は必ず含める
- タイトルは「どこで・何が」が一目で分かる形にする（例: 「注文履歴画面で注文一覧が表示されない」）。横断 issue はタイトル先頭に `【横断】` を付ける
- わからなかった項目は削除せず「不明」と書いて残す（開発者が聞き直すべき点が見えるようにするため）
- 「開発者向け参考情報」以外のセクションに技術用語・ファイルパスを書かない
- 報告者が自発的に述べた実現案は「報告者のアイデア（参考）」として区別して記録し、それ以外の解決策・実装方法・仕様案を書かない

### 対象領域の定型形式（必須）

「開発者向け参考情報」セクションの先頭に、次の定型形式で対象領域を必ず記載する。この記載は、以後の run で issue 索引ワーカーが照合に使う正本となる（記載が無い issue は推定にフォールバックされ、照合精度が落ちる）:

```
対象領域: <画面の言葉での領域> / <ディレクトリ・モジュール等の技術表現>
```

- 領域が複数ある場合は1領域1行で列挙する
- 例: `対象領域: 注文履歴画面 / src/features/order-history/`

### 更新履歴（必須）

本文の最終セクションに置き、登録・統合のたびに1行追記する:

```markdown
## 更新履歴
- YYYY-MM-DD: 初回登録（ケース1〜2）
- YYYY-MM-DD: ケース追加（[短い説明]）
```

## 単一ケース用の標準形式

グループに含まれるケースが1件のときに使う。

**不具合用:**

```markdown
## 概要
[1〜2行で問題の要約]

## 対象の画面・機能
[画面名・機能名。特定できなかった場合は「不明（再現手順を参照）」]

## 実際に起きたこと
[起きたこと。画面に表示されたメッセージがあれば原文のまま]

## 期待していた動作
[本来どうなるはずだったか]

## 再現手順
1. [操作を順番に]
（不明な場合は「不明」と記載）

## 環境・頻度
- 使用環境: [スマホ / パソコン / ブラウザ名 など。不明なら「不明」]
- 発生頻度: [毎回 / ときどき / 一度だけ。不明なら「不明」]
- 発生時期: [いつ頃から]

## 影響
[この問題で誰が何をできなくて困っているか]

## 開発者向け参考情報
対象領域: [画面の言葉での領域] / [技術表現]
[調査で特定した関連機能・処理など。特定できなければ対象領域の行のみ]

## 更新履歴
- YYYY-MM-DD: 初回登録
```

**要望用**（実現方法・仕様のセクションを意図的に置かない。仕様は開発者が検討する）:

```markdown
## 困りごと・背景
[いま何に困っているか・なぜ必要か]

## 目的
[実現すると何が良くなるか]

## 対象の画面・機能
[関連する既存の画面・機能。既存機能と関係しない場合は「新規」]

## 現在の回避策
[いまどう対処しているか。無ければ「なし」]

## 頻度・影響範囲
[どのくらいの頻度で困るか・誰が困っているか]

## 報告者のアイデア（参考）
[報告者が自発的に述べた実現案があれば記載。仕様の決定ではなく参考情報。無ければこのセクションごと省略]

## 開発者向け参考情報
対象領域: [画面の言葉での領域] / [技術表現]
[調査で分かった関連機能・類似の既存実装など]

## 更新履歴
- YYYY-MM-DD: 初回登録
```

## 複数ケース用の標準形式

グループに複数ケースが統合されたときに使う。各ケースのセクション構成は、種類（不具合 / 要望）に応じて単一ケース用の該当部分を継承する。

```markdown
## 概要
[この issue に含まれる課題の要約と、1つの issue にまとめている理由
（例: いずれも注文履歴画面に関係し、対応時に変更範囲が重なるため）]

## 対象領域（共通）
[グループ共通の対象領域を画面の言葉で]

---

## ケース1: [ケースの1行タイトル]

### 実際に起きたこと
[不具合系の場合。要望系なら「困りごと・背景」「目的」等の構成にする]

### 期待していた動作
...

### 再現手順
...

### 環境・頻度・影響
...

---

## ケース2: [ケースの1行タイトル]
...

---

## 開発者向け参考情報
対象領域: [画面の言葉での領域] / [技術表現]
[ケース横断で分かった関連機能・処理など]

## 更新履歴
- YYYY-MM-DD: 初回登録（ケース1〜2）
```

## 本文書き換え（統合）手順

`action: merge` のグループで、起草ワーカーが書き換え後の完成本文を作る手順。**既存の記載内容を削除・要約・言い換えしてはならない**（他の報告者・開発者の記述を含むため）。

1. `gh issue view <番号> --json title,body` で現行のタイトルと本文を取得する
2. 既存本文が**単一ケース形式**の場合: 複数ケース用の標準形式へ再構成する。既存の内容は「ケース1」としてセクション見出しを付け替えるだけで、本文の文言はそのまま保持する
3. 既存本文が**複数ケース形式**の場合: 既存の最終ケース番号の続き（ケースN+1〜）として新ケースを追加する
4. 既存本文が**本スキル以外の形式**（テンプレート不一致・自由記述）の場合: 既存本文を丸ごと保持したまま、その下に区切り線と「## 追加のケース（YYYY-MM-DD）」以降のセクションとして新ケースを追加する。既存部分の再構成はしない
5. 新ケースの内容は該当ケースの `investigation` / `hearing` から起こし、わからない項目は「不明」と書く
6. 「開発者向け参考情報」の「対象領域:」を再計算して更新する（既存の領域は残し、新ケースで加わった領域を追加する）
7. 「更新履歴」に1行追記する: `- YYYY-MM-DD: ケース追加（[短い説明]）`。更新履歴セクションが無い既存 issue の場合は新設する
8. タイトルは原則変更しない。統合により内容が広がってタイトルが実態と合わなくなった場合のみ、変更後タイトルを起草し、Phase 7 の登録計画に「タイトル変更」として明示して承認を得る

書き換え後の完成本文は `drafts/update-<issue番号>.md` に出力する。GitHub への反映（`gh issue edit <番号> --body-file`）は Phase 7 の承認後に実行ワーカーが行う。
