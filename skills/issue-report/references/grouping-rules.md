# issue-report: グルーピング規則

分類エージェント（縮退時は parent）が Phase 5 で適用する判定規則。目的は、**issue をワークツリー1本の分割単位として、並列開発でコンフリクトしない単位に編成する**ことである。

## 1. 想定変更領域（touch set）の定義

- 想定変更領域とは「この課題に対応するとき、変更が及ぶと見込まれる範囲」を**領域単位**で見積もったものである
- 粒度は **画面 / ルート / モジュール / ディレクトリ prefix** とし、**ファイル単位の変更予測はしない**。理由は2つ:
  1. 要望系は仕様が未決定（本スキルは仕様を決めない）であり、ファイル単位の予測は原理的に不可能
  2. ファイル単位まで見積もろうとすると実装方法の検討に踏み込み、「解決策・実装仕様を決めない」制約に抵触する
- **影響領域の見積もりは仕様の決定ではない**。「どの領域が変わりそうか」までで止め、「どう変えるか」には踏み込まない
- 各ケースの `touchAreas` では、ケースの主題である領域に `primary: true` を付け、対応の過程で触れる可能性があるだけの領域と区別する

## 2. 重なり判定と統合の原則

2つのケース（または、ケースと既存 issue）の想定変更領域が以下のいずれかに該当する場合、「干渉の可能性あり」と判定して**統合**する:

- 同一領域を含む（例: 双方が「注文履歴画面」を含む）
- 一方の領域が他方を包含する（例: 「注文機能全体」と「注文履歴画面」）
- 領域間に強い依存があり、片方の変更が他方の変更を誘発する見込みが高い

**判定はこう倒す: 干渉の可能性があれば統合する。** 分割しすぎて並列開発でコンフリクトする（誤って分ける）方が、まとめすぎて並列度が下がる（誤ってまとめる）よりも損失が大きい。確信が持てない場合は統合を選ぶ。

例外として、横断領域を介した重なりは次節の規則で扱う。

## 3. 推移的連結の遮断（横断領域の扱い）

**横断領域**とは、複数の画面・機能から共有される基盤を指す。例: 共通 UI コンポーネント、共通型定義、デザイントークン、ルーティング表、共通ユーティリティ、認証基盤。調査ワーカーは該当する `touchAreas` エントリに `crossCutting: true` を付ける。

横断領域を素朴に重なり判定へ含めると、「ケースA ─ 共通コンポーネント ─ ケースB」という間接的な重なり（推移的連結）で全ケースが1つの巨大 issue につながってしまう。これを次の2規則で遮断する:

- **規則1（除外）**: 通常ケース同士の重なり判定では、`crossCutting: true` の領域への接触を**重なりとして数えない**。ケースAとBが両方とも共通ボタンに触れる、というだけでは統合しない
- **規則2（昇格）**: ケースの**主対象そのもの**（`primary: true` の領域）が横断領域である場合（例: 「全画面のボタンのデザインを変えたい」）、そのケースは**横断 issue** として通常グループから分離する
  - 主対象が同一の横断領域である既存の横断 issue（または同じ run 内の横断ケース）があれば、そこへ統合する（同じ共有基盤を主対象とする課題同士は確実に干渉するため）
  - **横断 issue と通常 issue は統合しない**（どちらの方向にも）

## 4. 横断マーカーの運用

- **正本はラベル `scope:cross-cutting`**。横断 issue かどうかの判定は、このラベルの有無**のみ**で行う
- タイトル接頭辞 **`【横断】`** は人間が issue 一覧で識別するための表示であり、判定には使わない（「横断」「共通」等の語がタイトルに含まれていても、ラベルが無ければ横断 issue として扱わない）
- 新規の横断 issue では、分類エージェントが `grouping-plan.json` の `title` に `【横断】` 接頭辞を含めて確定し、実行ワーカーがラベル `scope:cross-cutting` を付与する。ラベルがリポジトリに存在しない場合は `gh label create` で作成する（コマンドは `orchestration-protocol.md` Phase 7-2）

## 5. 既存 issue 照合規則

- 照合対象は **open issue 全件**（`issue-index.json`）。closed issue は照合しない
- 各既存 issue の対象領域は、本文の「対象領域:」定型記載（[`issue-format.md`](issue-format.md)）を最優先で読む。定型記載が無い issue（本スキル以外で作られたもの等）は、タイトルと本文から領域を推定し `areaSource: "inferred"` とする
- ケースの想定変更領域と既存 issue の対象領域に §2 の重なりがあれば、そのケースは**既存 issue への統合候補**（`action: merge`）とする。ただし:
  - 横断ラベル付き issue への統合は、ケースの主対象が同一の横断領域である場合（§3 規則2）に限る
  - 通常ケースが横断領域への接触だけで既存の横断 issue と重なっても、統合しない（§3 規則1）
- 複数の既存 issue と重なる場合は、主対象（`primary`）の重なりが最も強い1件へ統合する
- 統合先が進行中かどうか（アサイン・リンクされた PR の有無）は考慮しない。仕組みをシンプルに保つため、重なりがあれば統合する

## 6. 出力に必ず含めるもの

分類エージェントは `grouping-plan.json` の各グループに以下を必ず書く:

- `rationale`: どの領域の重なり（またはどの規則の適用）で統合・分離したかの根拠。開発者向けの技術表現でよい
- `userExplanation`: parent がそのままユーザーへ提示できる、技術用語・ファイルパスを使わないグルーピング理由（例: 「1つ目と3つ目は同じ『注文履歴』の画面の仕組みに関係するため、1つの課題にまとめます」）
