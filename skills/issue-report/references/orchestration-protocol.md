# issue-report: オーケストレーション詳細プロトコル（Phase 0〜7）

parent（PM）と各ワーカーの詳細な役割・成果物ファイルのスキーマ・ゲート条件を定める。
グルーピングの判定規則は [`grouping-rules.md`](grouping-rules.md)、issue 本文の形式と書き換え手順は [`issue-format.md`](issue-format.md) を参照する。

## 実行ディレクトリと成果物ファイル

parent は開始時に、環境のスクラッチ領域（無ければ `/tmp`）配下へ実行ディレクトリ（例: `mktemp -d` で作成した `issue-report-XXXX`）を作成する。以後の全成果物はこの配下に置き、ワーカーへは**ファイルパスを渡す**（内容を parent のコンテキストへ展開しない）。

```
<run-dir>/
├── report.md                 # ユーザーの報告原文（引数＋自由発話を parent が保存）
├── cases.json                # 分解ワーカー出力
├── investigation-<id>.json   # 調査ワーカー出力（ケースごと）
├── hearing-<id>.json         # ヒアリング結果（parent が記録）
├── issue-index.json          # 索引ワーカー出力
├── grouping-plan.json        # 分類エージェント出力
└── drafts/
    ├── new-<groupId>.md      # 新規 issue の起草本文（本文のみ。タイトルは grouping-plan.json の title）
    └── update-<issue番号>.md # 統合先 issue の書き換え後完成本文（本文のみ）
```

### cases.json

```json
{
  "cases": [
    {
      "id": "c1",
      "kind": "bug",
      "summary": "注文履歴画面で一覧が表示されない（画面の言葉での1行要約）",
      "excerpt": "報告原文のうちこのケースに該当する部分の抜粋"
    }
  ]
}
```

- `kind`: `bug`（不具合系）| `request`（要望系）。分解時点の推定でよい
- `id` は `c1`, `c2`, … の連番

### investigation-<id>.json

```json
{
  "caseId": "c1",
  "kind": "bug",
  "screenCandidates": [
    { "label": "注文履歴画面", "summary": "過去の注文を一覧で見る画面" }
  ],
  "touchAreas": [
    {
      "area": "注文履歴画面",
      "paths": ["src/features/order-history/"],
      "primary": true,
      "crossCutting": false
    }
  ],
  "confidence": "high",
  "missingInfo": [
    { "item": "再現手順", "critical": true },
    { "item": "発生頻度", "critical": false }
  ]
}
```

- `screenCandidates`: ユーザーに提示する対象候補（最大5件、画面の言葉）
- `touchAreas`: 想定変更領域。粒度は領域単位（画面 / ルート / モジュール / ディレクトリ prefix）とし、ファイル単位の変更予測はしない（[`grouping-rules.md`](grouping-rules.md)）。`primary` はケースの主対象、`crossCutting` は横断領域（共有コンポーネント等）への接触
- `confidence`: `high` | `low`。対象特定の確度（Phase 3 のトリアージに使う）
- `missingInfo`: 開発者がそのまま動くために不足している情報。`critical: true` は開発着手を左右する不足

### hearing-<id>.json

```json
{
  "caseId": "c1",
  "confirmedTarget": "注文履歴画面",
  "answers": [
    { "item": "再現手順", "answer": "注文一覧を開いて並び替えを押すと発生" },
    { "item": "発生頻度", "answer": "不明" }
  ],
  "reporterIdea": null
}
```

- `confirmedTarget`: Phase 3 でユーザーと合意した対象（画面の言葉）。特定できなかった場合は `"対象不明"`
- ユーザーが「わからない」と答えた項目は `"不明"` と記録し、それ以上追わない
- `reporterIdea`: 報告者が自発的に述べた実現案があれば記録（深掘りしない）

### issue-index.json

```json
{
  "issues": [
    {
      "number": 12,
      "title": "注文履歴画面で注文一覧が表示されない",
      "labels": ["bug"],
      "crossCutting": false,
      "areas": ["注文履歴画面 / src/features/order-history/"],
      "areaSource": "declared"
    }
  ]
}
```

- `crossCutting`: ラベル `scope:cross-cutting` の有無（ラベルのみで判定。タイトル文言では判定しない）
- `areaSource`: `declared`（本文の「対象領域:」定型記載から取得）| `inferred`（定型記載が無く本文から推定）

### grouping-plan.json

```json
{
  "groups": [
    {
      "id": "g1",
      "action": "new",
      "cases": ["c1", "c3"],
      "targetIssue": null,
      "crossCutting": false,
      "title": "注文履歴画面の一覧表示と並び替えの不具合",
      "areas": ["注文履歴画面 / src/features/order-history/"],
      "rationale": "c1 と c3 は主対象領域が同一（注文履歴画面）で干渉の可能性があるため統合",
      "userExplanation": "1つ目と3つ目は同じ「注文履歴」の画面の仕組みに関係するため、1つの課題にまとめます"
    },
    {
      "id": "g2",
      "action": "merge",
      "cases": ["c2"],
      "targetIssue": 12,
      "crossCutting": false,
      "title": null,
      "areas": ["注文履歴画面 / src/features/order-history/"],
      "rationale": "既存 #12 と対象領域が重なるため統合",
      "userExplanation": "2つ目は、すでに登録されている課題と同じ場所の話なので、その課題に書き加えます"
    }
  ],
  "summary": { "newCount": 1, "mergeCount": 1, "crossCuttingCount": 0 }
}
```

- `action`: `new`（新規登録）| `merge`（既存 issue へ本文統合）
- `title`: 新規グループの issue タイトル（「どこで・何が」形式。横断グループは分類エージェントが `【横断】` 接頭辞を含めて確定する）。`merge` グループは `null`
- `userExplanation`: parent がユーザーへ提示する、技術用語を使わないグルーピング理由（分類エージェントが必ず書く）

## ユーザー対話のスケール設計

ユーザー対話は parent のみが行い、**対話の回数がケース数に比例しない**よう次の方式を全フェーズで守る:

- **一括確認**（Phase 1・3・7）: ケースやグループの一覧を表で提示し、「違うもの・足りないものはありますか？」の形で1回にまとめる
- **確度トリアージ**（Phase 3）: 確度 `high` のケースは表で一括確認し、`low` のケースだけ個別に対話する
- **不足情報駆動**（Phase 4）: 質問セットを全ケースへ機械的に適用せず、調査ワーカーが返した `missingInfo` のうち `critical` なものから優先して質問する。質問は利用可能な質問機能で一度に最大4問・回答例つき・**「わからない」選択肢必須**。ケース数が多く全件を聞くと負担が大きい場合は、`critical` な不足だけに絞り、残りは「不明」と記載する
- **計画単位承認**（Phase 7）: 承認は登録計画全体に対して1回。個別本文はユーザーが希望したときだけドリルダウン表示する

## Phase 0: 受け取り

parent が行う:

1. 実行ディレクトリを作成する
2. 引数の内容（あれば）と自由発話を `report.md` に保存する。引数が空の場合は「どんな問題や、こうしてほしいという要望がありますか？いくつあっても大丈夫です」と自由に話してもらう
3. 進捗チェックリスト（Phase 1〜7）を応答に貼って進捗を追跡する

## Phase 1: ケース分解

**分解ワーカーへの依頼**: `report.md` のパスを渡し、独立した困りごと・要望ごとに分割して `cases.json` を出力させる。

- 分割の単位は「報告者の言葉として独立している困りごと・要望」。この段階で課題同士をまとめてはならない（統合の判断は Phase 5 の分類エージェントの責務）
- `kind`（bug / request）はこの時点の推定でよい
- 報告が1件しか含まない場合も `cases.json`（1要素）を作り、以降のフェーズは同じ流れで進む

**parent の一括確認**: `cases.json` のサマリー（id・種別・1行要約）を表でユーザーに提示し、分け方の認識を合わせる:

```markdown
## 📋 お話の整理

お話には [N] 件の別々の困りごと・要望が含まれているようです：

| No | 種類 | 内容 |
|----|------|------|
| 1 | 不具合 | [画面の言葉での1行要約] |
| 2 | 要望 | [画面の言葉での1行要約] |

この分け方で合っていますか？（まとめて1つの話だった・他にもある、などあれば教えてください）
```

**Gate**: 分解の確認が取れるまで Phase 2 へ進まない。修正の指摘があれば分解ワーカーへ差し戻して `cases.json` を更新する。

## Phase 2: 並列実態調査 ∥ issue 索引作成

parent は次の2系統を**同時並行**で起動し、完了を待つ。parent 自身はコード・issue の中身を読まない。

**調査ワーカー（ケースごとに1体・並列・読み取り専用）への依頼**: 担当ケース1件分（`cases.json` の該当エントリ）を渡し、現在のリポジトリを読み取り専用で調査して `investigation-<id>.json` を出力させる。

- ユーザーの言葉は画面の言葉であるため、UI に表示される文言（ボタンラベル・見出し・メッセージ）でのテキスト検索が最も有効
- 画面・ルーティング・主要機能のディレクトリ構造も確認する。セマンティック検索が利用可能なら併用する
- 想定変更領域（`touchAreas`）は領域単位で見積もり、主対象（`primary`）と横断領域への接触（`crossCutting`）を区別する（判定基準は [`grouping-rules.md`](grouping-rules.md)）
- ソースコード・設定ファイルの編集は禁止（出力は成果物 JSON のみ）

**issue 索引ワーカー（1体）への依頼**: `gh issue list --state open` を全件取得し（`--limit` を十分大きく指定する）、各 issue の本文から対象領域を抽出して `issue-index.json` を出力させる。

- 対象領域は本文の「対象領域:」定型記載（[`issue-format.md`](issue-format.md)）を最優先で読み、無い issue は本文テキストから推定して `areaSource: "inferred"` を付ける
- `crossCutting` はラベル `scope:cross-cutting` の有無のみで判定する

## Phase 3: 対象の特定確認

parent が `investigation-<id>.json` の `confidence` でトリアージする:

**確度 high のケース**: 一覧表で一括確認する。

```markdown
## 🔍 おっしゃっている場所の確認

調べた結果、それぞれ次の画面・機能のことだと思われます：

| No | お話 | 関係する画面・機能 |
|----|------|------------------|
| 1 | [1行要約] | [画面名]（[ユーザーから見た1行説明]） |

違っているものがあれば番号で教えてください。
```

**確度 low のケース**: 個別に候補（最大5件、画面の言葉）を提示して確認する。候補が見つからない・ユーザーが特定できない場合は、「その画面を開くまでに、どのメニューやボタンを押しますか？」のように操作の流れを一歩ずつ聞いて絞り込む。追加の手がかりが得られたら該当ケースの調査ワーカーを再起動してよい。それでも特定できない場合は「対象不明」のまま先へ進んでよい（開発者が後から特定できるだけの情報を Phase 4 で集める）。

この段階ではファイルパス・技術用語をユーザーに見せない。

**Gate**: ケースごとに課題の意味と場所の認識（「おっしゃっている課題は◯◯という意味で、場所は◯◯で合っていますか？」）を合わせるまで Phase 4 へ進まない。ただし「特定できなかった」という合意でもよい。確認結果（対象の確定・変更・対象不明）は parent が `investigation-<id>.json` に追記せず、`hearing-<id>.json` 側に記録する。

## Phase 4: 課題のヒアリング

parent が `missingInfo` を束ね、不足している情報だけを質問する。質問セットは報告の種類で分ける（各項目は不足している場合のみ聞く）:

**不具合系（kind: bug）:**

| 聞くこと | 質問の例 |
|---|---|
| 実際に起きたこと | 何が起きましたか？（画面に出たメッセージがあれば、そのままの文で教えてください） |
| 期待していたこと | 本当ならどうなるはずだと思っていましたか？ |
| 再現手順 | どんな操作をしたら起きましたか？（順番に。途中までしか覚えていなくても大丈夫です） |
| 環境・頻度 | いつ頃から起きていますか？ 毎回起きますか？ スマホとパソコンのどちらで使っていましたか？ |
| 影響 | この問題で、いま何ができなくて困っていますか？ |

「期待していたこと」と「実際に起きたこと」は必ず分けて記録する（混ざっていると、開発者が仕様なのか不具合なのか判断できない）。

**要望系（kind: request）:**

| 聞くこと | 質問の例 |
|---|---|
| 困りごと | いま、何に困っていますか？ どこが不便ですか？ |
| 目的 | それができると、何が良くなりますか？（例: 作業時間が減る、ミスが減る、お客様に説明しやすくなる） |
| 現在の回避策 | いまはどうやって対処していますか？ |
| 頻度・影響範囲 | どのくらいの頻度で困りますか？ 同じことで困っている人は他にもいますか？ |

**要望系で聞かないこと**: どんな画面・ボタン・仕様にしてほしいかという実現方法。仕様の検討は開発者の工程であり、非エンジニアに仕様の判断を委ねることはかえってリスクになる。ユーザーが自発的に実現案を話した場合は `reporterIdea` として記録するだけにとどめ、深掘りしない。

回答（「不明」を含む）は parent が `hearing-<id>.json` に記録する。

## Phase 5: グルーピング・統合判定

**分類エージェント（1体）への依頼**: `investigation-*.json` 全件・`hearing-*.json` 全件・`issue-index.json` のパスと、[`grouping-rules.md`](grouping-rules.md) への参照を渡し、`grouping-plan.json` を出力させる。

分類エージェントが行うこと:

- 各ケースの `touchAreas` と既存 issue の `areas` の重なり判定（既存 issue と重なるケースは `action: merge`）
- 新規ケース同士の重なり判定（重なるケースは同一グループへ統合）
- 横断判定（主対象が横断領域のケースの昇格・分離）
- グループごとの `rationale`（開発者向けの根拠）と `userExplanation`（画面の言葉での説明）の記述

parent は結果の意味判断をしない。`summary` の件数と構成だけ把握し、Phase 6 へ進む。

## Phase 6: 本文起草

**起草ワーカー（グループごとに1体・並列）への依頼**: 担当グループ1件分の材料（`grouping-plan.json` の該当グループ、含まれるケースの `investigation-*.json` / `hearing-*.json` のパス）を渡し、[`issue-format.md`](issue-format.md) に従って本文を起草させる。

- `action: new` → 本文のみを `drafts/new-<groupId>.md` に出力する（タイトルは `grouping-plan.json` の `title` を正とする）
- `action: merge` → 起草ワーカー自身が `gh issue view <番号> --json body,title` で現行本文を取得し、書き換え後の**完成本文**を `drafts/update-<issue番号>.md` に出力する（書き換え手順は [`issue-format.md`](issue-format.md)。この時点では GitHub へ書き込まない）。統合によりタイトル変更を提案する場合は、完了報告で変更後タイトルを parent へ返し、parent が登録計画に「タイトル変更」として含める
- 対象リポジトリの `.github/ISSUE_TEMPLATE/` を確認し、該当する種類のテンプレートがあればその構成を優先する
- わからなかった項目は削除せず「不明」と書いて残す

## Phase 7: 登録計画の承認と実行

### 7-1: 登録計画の提示と承認

parent が登録計画をサマリー表で一括提示する:

```markdown
## 📝 登録内容の確認

以下の内容で GitHub に登録・更新します：

| No | タイトル | 含まれるお話 | 登録方法 |
|----|---------|------------|---------|
| 1 | [タイトル] | No.1, No.3 | 新しく登録 |
| 2 | （既存の課題「[既存タイトル]」） | No.2 | すでにある課題に書き加え |

- [グループごとの userExplanation を箇条書きで]

それぞれの詳しい文面を見たい場合は番号で教えてください。この内容で登録してよいですか？
```

- 承認は計画単位で1回。ドリルダウン要望があれば該当 draft の本文を提示する
- 修正の要望があれば、該当グループの起草ワーカー（必要なら分類エージェント）へ差し戻して反映し、再提示する

**Gate**: 明確な承認を得るまで実行ワーカーを起動しない。

### 7-2: 登録実行

**実行ワーカー（1 issue 操作 = 1 packet・並列）への依頼**: 承認済みの packet 1件（draft ファイルのパス・操作種別・統合先番号・ラベル指示のみ）を渡す。計画全体や他グループの情報を渡してはならない。

- `action: new`: `gh issue create --title "<grouping-plan の title>" --body-file <draft>` で登録する。ラベルは `gh label list` で既存ラベルを確認し、bug / enhancement 相当のものがあれば付与する（無ければラベルなしで登録する）
- `action: new` かつ `crossCutting: true`: 追加でラベル `scope:cross-cutting` を付与する（`【横断】` 接頭辞は分類エージェントが `title` に含めて確定済みのため、実行ワーカーはタイトルへ手を加えない）。ラベルが存在しなければ `gh label create scope:cross-cutting --description "横断領域（共有基盤）を主対象とする issue。他の issue と統合しない" --color 5319E7` で作成してから付与する
- `action: merge`: `gh issue edit <番号> --body-file <draft>` で本文を書き換える（タイトルは原則変更しない。変更が承認済み計画に含まれる場合のみ `--title` を併用する）
- 操作結果（issue URL・成功/失敗）を parent へ返す

### 7-3: 完了報告

parent が登録・更新した issue の URL 一覧をユーザーに報告する。失敗した packet があれば、失敗内容と再試行または手動登録の案内を平易な言葉で伝える。

**gh コマンドが使えない・認証されていない場合**: エラーで止めず、承認済みの各 draft（タイトルと本文）をコピーできる形で提示し、手動登録の手順（GitHub でこのプロジェクトのページを開く → Issues タブ → New issue → 貼り付けて Submit）を平易な言葉で案内する。
