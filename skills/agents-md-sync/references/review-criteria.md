## 1. レビューサブエージェントへの入力

各レビューエージェントには次を渡す。**執筆エージェントの下書きの経緯・会話は渡さない**（fresh context を保つ）。

- 審査対象の AGENTS.md 全文とそのパス
- 確定済みの親 AGENTS.md 全文（ルート審査時は不要）
- 対象ディレクトリへの読み取りアクセス（spot-check 用）

## 2. 審査項目

上から順に検査する。1つでも該当箇所があれば FAIL（修正指示付き）。

1. **重複**: 親・祖先の AGENTS.md に既にある記述 → 削除指示
2. **実在性 spot-check**: 記載されたコマンド・パス・シンボルから**最低5件**（5件未満しか無ければ全件）を無作為に選び、実際にリポジトリ内で確認する。存在しない・動かないものがあれば修正指示。陳腐化した構造参照は無いより有害（エージェントを積極的に誤誘導する）ため、この項目が最重要
3. **根拠のない断定**: コード上の根拠を確認できない断定的記述 → 〔要確認: 理由〕マーカー化の指示
4. **一般論の混入**: どのプロジェクトにも当てはまる訓辞・アドバイス（「テストを書け」「clean code」等） → 削除指示
5. **禁止のみの記述**: 代替を伴わない禁止事項 → 代替とペア化する指示（代替が不明なら〔要確認〕化）
6. **形式**: 一律 1000 行以下 / OVERVIEW 節あり / 生成メタデータ行あり / 電報体（前置き・冗長な丁寧語がない）

補足: ナビゲーション情報（STRUCTURE・WHERE TO LOOK・CODE MAP）は、**正確であれば**知識ベース型の正当な内容であり削らない。削る対象は「重複・不正確・一般論」であって「情報量」ではない。

## 3. 判定の出力形式

```
判定: PASS | FAIL
[FAIL の場合]
- L{行番号}: {問題の種別} — {具体的な修正指示}
- ...
```

- 修正指示は執筆エージェントの新インスタンスがそのまま実行できる具体性で書く
- 書き直しは1回まで。再レビューでも FAIL の箇所は〔要確認〕マーカー化して人間に引き継ぐ

## 4. drift 判定基準（update モード）

scan-repo.sh が生成メタデータ行の `commit=` を基準に判定する。既定の閾値:

| 判定 | 条件 |
|---|---|
| NEEDS-UPDATE | 記録コミット以降、対象ディレクトリで **20 ファイル以上**変更、または追跡ファイル総数の **30% 以上**が変更 |
| NO-METADATA | 生成メタデータ行が無い（手書き・他ツール生成・記録コミットが履歴に無い）→ 更新候補として扱う |
| BRIDGE-MISSING | AGENTS.md はあるが同階層に CLAUDE.md ブリッジが無い → Phase 5 で作成 |
| OK | 上記いずれにも該当しない → スキップ（冪等・低コスト） |

- 閾値（20 ファイル / 30%）は「数ファイルの変更のたびに全文更新すると運用コストが利得を上回る」ための足切りであり、実測に基づく最適値ではない。運用観察で調整してよい。**変更するときは scan-repo.sh 内の定義と本表を同時に更新する**
- 閾値未満でもユーザーがパス限定引数で明示した階層は更新対象にする
