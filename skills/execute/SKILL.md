# execute - タスク遂行オーケストレーション

複雑なタスクを適切に整理・分割し、利用可能な委譲機構（サブエージェント・タスク委譲など、実行環境が提供するもの）を効果的に活用して、実装や調査を遂行するためのスキル。呼び出し元はプロジェクトマネージャーとして全体を統括し、各タスクを最適な形で実行する。

## 追加指示の扱い

このスキルの起動時にユーザーから追加指示が渡された場合、その追加指示には次の情報が含まれる可能性が高い。

- 実行したいタスクの詳細な説明
- 対象ファイルやディレクトリのパス
- 実装における制約条件や要件
- 参照すべき既存コードや設計方針

追加指示は任意とし、渡されていない場合は本スキルの標準フローのみを実行する。追加指示が渡された場合は、本スキル内の他の方針よりも優先して適用する。ただし後述の「重要な制約」に反する追加指示の場合は、直ちに停止してユーザーに確認を求めること。

## 基本方針

- **プロジェクト統括**: 実装詳細よりも全体の整理・統括に注力する
- **委譲の活用**: 実作業は、詳細な指示のもとで利用可能な委譲機構に依頼する
- **タスク分割**: 大きすぎるタスクは適切な粒度に分割して整理する
- **並列実行**: 依存関係を整理し、並列可能なタスクは複数の委譲先で同時に実行する
- **詳細な指示**: 各委譲先への指示は具体的かつ明確に記述する
- **集中実行**: 与えられた指示に集中し、無駄な作業は避ける
- **有意義なコメント**: 過去の実装を示唆する不要なコメントは避け、真に意味のあるコメントのみを残す

## 委譲先の選択

実行環境が委譲機構（サブエージェント・タスク委譲・並列ワーカーなど）を提供している場合は、次の優先順位で委譲先を選ぶ。委譲機構が存在しない環境では、呼び出し元自身が各タスクを順に実行する。

1. **プロジェクト固有の委譲先（最優先）**
   - プロジェクトに専用の委譲先（カスタムエージェント・専用ワーカー等）が定義されている場合、まずその存在と内容を確認する
   - プロジェクトのコーディング規約やアーキテクチャに精通した委譲先が、タスクの性質に合致する場合は、必ずそれを使う
2. **汎用の委譲先（フォールバック）**
   - プロジェクト固有の委譲先が存在しない、または適切なものがない場合のみ、実行環境が提供する汎用の委譲先（汎用実行・コードベース探索・実装計画立案など）を使う

> プロジェクト固有の委譲先がどこに、どの形式で定義されるかは実行環境により異なる。特定の設定ディレクトリやファイルパスを前提として固定せず、その環境で委譲先を列挙・確認する手段を用いて把握すること。

## 実行手順

### 1. タスクの分析と理解

- タスクの目的と範囲を明確化する
- 必要な情報を収集する（コード調査、ドキュメント確認等）
- 技術的制約と依存関係を把握する

### 2. タスクの整理と分割

**タスク分割基準:**

- 各タスクは独立して実行可能な単位に分割する
- 1タスクは1〜2時間程度で完了できる粒度を目安にする
- タスク間の依存関係を明確化する

**依存関係の整理:**

- 必ず順次実行すべきタスクを特定する
- 並列実行可能なタスクをグループ化する
- クリティカルパスを識別する

### 3. 委譲先への指示作成

**指示に含めるべき内容:**

- タスクの目的と背景
- 具体的な実装/調査内容
- 技術仕様や制約事項
- 完了条件
- 活用すべきツールや MCP の明示（利用可能な場合）

**指示の例:**

```
タスク: ユーザー認証APIの実装

目的:
- JWT認証を使用したログイン/ログアウトAPIの実装

実装内容:
1. /api/auth/login エンドポイントの作成
2. /api/auth/logout エンドポイントの作成
3. JWT トークン生成・検証ロジックの実装
4. 既存のミドルウェアとの統合

技術仕様:
- 使用ライブラリ: jsonwebtoken
- トークン有効期限: 24時間
- リフレッシュトークンは不要

完了条件:
- APIが正常に動作すること
- 単体テストが追加され、すべてパスすること

ツール活用（利用可能な場合）:
- セマンティック検索/LSP 系 MCP: 既存の認証関連コードを検索して設計を理解する
- ライブラリドキュメント MCP: jsonwebtoken の最新ドキュメントを確認する
```

### 4. タスクの実行管理

**順次実行タスク:**

- 依存関係のあるタスクは順番に実行する
- 前のタスクの完了を確認してから次へ進む

**並列実行タスク:**

- 独立したタスクは複数の委譲先で同時に実行する
- 各委譲先の進捗を追跡する

### 5. 進捗の追跡と統合

- 各タスクの完了状況を確認する
- 実装結果をレビューして統合する
- 全体の整合性を検証する

### 6. 最終確認と報告

**確認項目:**

- すべてのタスクが完了していること
- 実装が要件を満たしていること
- テストが正常に通ること
- ドキュメントが更新されていること（必要な場合）

**報告構成:**

- 📋 実行したタスクの概要
- ✅ 完了した内容
- 🔧 実装の詳細
- 🧪 テスト結果
- 📝 注意事項や今後の課題
- 📊 **完成度**: [0-100] タスク完了度
- 📊 **品質**: [0-100] 実装品質

## 活用するツール・MCP

実作業では、実行環境が提供する基本ツール（ファイルの読み書き・コード検索・コマンド実行など）を活用する。加えて、以下のような MCP が利用可能な場合は積極的に活用し、委譲先への指示にもその活用を明示する。

- **セマンティック検索 / LSP 系 MCP（例: serena）**: セマンティック検索、シンボル検索、ドキュメント化。効率的な調査のために活用する
- **ライブラリドキュメント MCP（例: context7）**: フレームワーク・ライブラリの最新ドキュメント参照。API 仕様の確認に活用する
- **ブラウザ操作 MCP（例: playwright）**: ブラウザ操作、DevTools のエラー確認、ページ表示確認に活用する

## 委譲先への指示のポイント

1. **MCP の積極利用を明示**: 利用可能な MCP を使うよう明確に指示する
2. **詳細な仕様提供**: 技術的な詳細や制約を漏れなく伝える
3. **完了条件の明確化**: 何をもって完了とするかを具体的に定義する
4. **コンテキスト共有**: プロジェクト全体の中での位置づけを説明する
5. **品質基準の提示**: テスト、コード品質、ドキュメント等の基準を明示する

## 重要な制約

- タスク分割は適切な粒度で行う（大きすぎず小さすぎず）
- 委譲先への指示は具体的かつ詳細にする
- 並列実行可能なタスクは積極的に並列化する
- 各タスクの依存関係を明確に管理する
- 無駄な作業や不要なコメントは避ける
- 全体の進捗と品質を常に把握する
