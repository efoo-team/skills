# execute - タスク遂行オーケストレーション

複雑なタスクを整理・分割し、利用可能な委譲機構（サブエージェント・タスク委譲など、実行環境が提供するもの）を統括して遂行する。呼び出し元はプロジェクトマネージャーとして全体を統括する。

## 追加指示の扱い

起動時にユーザーから追加指示（実行したいタスクの詳細、対象パス、制約条件、参照すべき設計方針など）が渡された場合、本スキルの他方針より優先して適用する。ただし委譲による統括・順次タスクの完了確認といった本スキルの前提を壊す追加指示は、直ちに停止してユーザーに確認する。渡されない場合は標準フローのみを実行する。

## 委譲先の選択

実行環境が委譲機構（サブエージェント・タスク委譲・並列ワーカーなど）を提供している場合は、次の優先順位で選ぶ。委譲機構が無い環境では、呼び出し元自身が各タスクを順に実行する。

1. **プロジェクト固有の委譲先（最優先）**: プロジェクトのコーディング規約やアーキテクチャに精通した専用の委譲先が定義され、タスクの性質に合致する場合は、必ずそれを使う。
2. **汎用の委譲先（フォールバック）**: プロジェクト固有の委譲先が無い、または適切でない場合のみ、実行環境が提供する汎用の委譲先を使う。

> プロジェクト固有の委譲先がどこにどの形式で定義されるかは実行環境により異なる。特定の設定ディレクトリやパスを前提に固定せず、その環境で委譲先を列挙・確認する手段で把握する。

## 実行手順

進捗を追跡できるよう、次のチェックリストを応答に貼って進める。

- [ ] タスクの目的と範囲を明確化し、技術的制約と依存関係を把握する
- [ ] 独立して実行可能な単位へ分割し、タスク間の依存関係を整理する
- [ ] 順次実行すべきタスクと並列実行できるタスクを切り分け、クリティカルパスを識別する
- [ ] 各委譲先への指示を作成する（下記テンプレートに従い、活用すべきツール・MCP を明示する）
- [ ] 独立タスクは並列に委譲し、依存タスクは順に委譲する
- [ ] 各タスクの完了を確認し、結果を統合して全体の整合性を検証する
- [ ] 「完了前セルフレビュー」に答えてから報告する

**Gate（順次タスク）**: 依存関係のあるタスクは、前タスクの完了を確認するまで次のタスクへ進まない。

## 委譲先への指示テンプレート

各委譲先への指示は、次の見出しを埋めて渡す。技術仕様・制約・完了条件を漏れなく伝え、プロジェクト全体での位置づけも共有する。

```
目的:
実装内容:
技術仕様:
完了条件:
ツール活用（利用可能な場合）:
```

## 活用するツール・MCP

実作業では実行環境の基本ツール（ファイル読み書き・コード検索・コマンド実行）に加え、利用可能な MCP を積極的に活用し、委譲先への指示にもその活用を明示する。

- セマンティック検索 / LSP 系（例: serena）: シンボル検索・調査
- ライブラリドキュメント（例: context7）: フレームワーク・ライブラリの最新 API 仕様の確認
- ブラウザ操作（例: playwright）: 表示確認・DevTools のエラー確認

## 報告

- 📋 実行したタスクの概要
- ✅ 完了した内容 / 🔧 実装の詳細 / 🧪 テスト結果
- 📝 注意事項・今後の課題

## 完了前セルフレビュー

報告を確定する前に、次の3問に文章で答え、その回答を報告の末尾に含める（採点が目的ではなく、出力を確定する前に自身の行いを客観視するための強制点である）。

1. このタスクで確認できなかった事実・未検証の前提は何か
2. ユーザーの想定・期待に反する事実を見つけたのに、書いていないものはないか（あれば今、本文に追記する）
3. 出力のうち、事実ではなく推測に基づく箇所はどこか
