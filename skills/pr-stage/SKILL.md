# pr-stage - ステージ済み変更のコミット&PR作成

`git add` で既にステージング済みの変更を確認し、適切なコミットメッセージでコミットを作成し、Pull Request 作成までを、AI が誤操作しないことを最優先に実行するスキル。

## 明示起動限定

このスキルはコミット・push・PR 作成という不可逆かつ外部（リモート・GitHub）に影響する副作用を実行する。ユーザーが明示的に `/pr-stage`（Codex では `$pr-stage`）を起動した場合のみ実行する。会話の流れから自動発動してはならない。

## 目的

- 既にステージ済みの変更を、追跡性の高い comprehensive なコミットとしてまとめる
- 保護ブランチ（develop / main / master）への直接コミットを避け、必要ならフィーチャーブランチを自動作成する
- 手順のばらつきをなくし、再現性の高い PR 作成を行う

## 前提: プロジェクト規約への準拠

実行前に、リポジトリの規約ファイル（`CLAUDE.md` / `AGENTS.md` 等）を必ず確認する。記載されたブランチ運用ルール・使用言語・報告規則・ツール利用方針を遵守し、本スキルの指示と矛盾する場合は規約ファイルを優先する。規約ファイルにベースブランチの言及や運用ルールがなく判断できない場合は、直ちに操作を中止し、ユーザーに確認を求める義務を負う。

## 追加指示・引数の扱い

`argument-hint` の引数（`[commit-message]`）または追加指示がある場合は、本スキル内の自動判定より優先して適用する。追加指示には次の情報が含まれる可能性が高い。

- コミットメッセージ / タイトル
- PR タイトルや説明に含めたい情報
- ベースブランチの指定

引数が空の場合は下記の自動判定ルールで標準フローを進める。ただし「注意事項」に反する指定は受け付けず、直ちに停止して確認を求める。

## 前提条件

- Git リポジトリであること
- GitHub CLI (`gh`) がインストール・認証済みであること
- 変更が既に `git add` でステージングされていること
- 適切なブランチ権限があること

## 注意事項

- **ステージ前の確認必須**: このスキルを実行する前に、必ず `git add` で必要なファイルのみをステージングしておくこと
- **不要なファイルの除外**: 実行前に `git status` で確認し、関係のないファイルが含まれていないかチェックする
- **保護ブランチでの自動分岐**: `develop` / `main` / `master` にいる場合は、コミット前に自動でフィーチャーブランチを作成し、作業完了時に報告する
- **PR 作成時**: ベースブランチは自動判定（`develop` → `main` → `master` の順で存在確認）

## 実行フロー

### 1. ステージング状態の確認

以下のコマンドで現在の状態を確認する。

```bash
git diff --staged --name-only
git status
git branch --show-current
git log --oneline -10
```

### 2. ステージ済み変更の詳細確認

```bash
git diff --staged
```

ステージされている全ての変更内容を確認し、コミットメッセージの生成に活用する。

### 3. コミットメッセージの生成または使用

- **引数あり**: 引数をコミットタイトルとして使用
- **引数なし**: ステージ済み変更を分析して comprehensive（包括的）なコミットメッセージを自動生成
  - 変更内容から適切なプレフィックス（`feat` / `fix` / `docs` / `refactor` 等）を判定
  - 1 行タイトルだけでなく本文（body）を含め、Why → What → How の文脈を含める
  - 固定テンプレートにはせず、シニアエンジニアが書く洗練されたコミットとして、状況に応じて最適な構成・表現を選ぶ
  - 将来のリリースノート生成に貢献する品質を目指す

### 4. ブランチの確認とフィーチャーブランチ作成

**コミットの前に**現在のブランチを確認する。現在のブランチが `main` / `master` / `develop` の場合は、コミットする前に**必ずフィーチャーブランチを作成**する。保護ブランチへの直接コミットは行わない。

1. ステージ済み変更の内容を分析し、改修の主目的を特定する
2. 目的に即したブランチ名を決定する（例: `feature/add-user-authentication`, `fix/login-validation-error`）
3. 新しいブランチを作成してチェックアウトする

```bash
git checkout -b <決定したブランチ名>
```

**重要**: ブランチを自動作成した場合、全ての作業完了時に以下を必ずユーザーに報告する。

- 作成したブランチ名
- そのブランチで行った変更の概要

### 5. コミット実行

決定した comprehensive なコミットメッセージでコミットを作成する。`develop` / `main` / `master` への直接コミットは禁止であり、コミットは必ずフィーチャーブランチで行う（保護ブランチにいた場合はステップ 4 で作成済み）。

```bash
git commit -m "$(cat <<'EOF'
<commit-title>

<comprehensiveな本文>
EOF
)"
```

### 副作用実行前の確認（push / PR 作成の直前・必須）

push と PR 作成はリモートおよび GitHub に影響する不可逆操作である。実行の直前に、必ず以下をすべて確認する。

- 現在のブランチがフィーチャーブランチであり、`develop` / `main` / `master` でないこと
- ベースブランチが `develop` → `main` → `master` の判定ルールに適合していること（develop があれば必ず develop）
- コミット内容に無関係な差分が含まれていないこと
- 規約ファイル（CLAUDE.md / AGENTS.md 等）のブランチ運用ルールに反していないこと

いずれか 1 つでも満たされない、または規約ファイルにベースブランチの記載がなく判断できない場合は、push / PR を実行せず即時停止してユーザーに確認する。

### 6. リモートへプッシュ

```bash
git push -u origin <branch-name>
```

リモートブランチが存在しない場合は `-u` オプションで新規作成する。

### 7. Pull Request 作成

GitHub CLI を使用して PR を作成する。ベースブランチは `develop` → `main` → `master` の順で存在確認し、最初に見つかったブランチを指定する。タイトルと本文は日本語で記述する。

**必ず HEREDOC 形式を使用すること**（改行が `\n` リテラルになる問題を防ぐため）。

```bash
gh pr create --base <ベースブランチ> --title "PRタイトル" --body "$(cat <<'EOF'
ここに本文を記述する。
改行はそのまま反映される。

EOF
)"
```

**絶対にやってはいけない形式:**

```bash
# NG: 改行が \n リテラルになる
gh pr create --body "背景・課題\n原因\n対策"
```

## PR 文章の作成: github-pull-request スキルへ委譲

**PR のタイトルおよび本文の作成は必ず `github-pull-request` スキルに委譲する**。自力で文章を組み立ててはならない。`github-pull-request` をロードし、そのスキルの実行フローに従って PR 本文を生成する。

その上で、以下の**記載内容の網羅チェック**を必ず行う。目的は、PR 本文の各セクションの中身が中途半端にならないこと（一部だけ記載して他を漏らさないこと）。

1. **同一カテゴリ内の漏れ確認**: 記載するセクションについて、同一カテゴリの全要素が揃っていることを確認する
   - API 変更: 追加・変更・削除されたエンドポイントを全て記載。1 つだけ書いて他を省略しない
   - データ構造: 関連するエンティティ・カラム変更を全て記載
   - フロントエンド変更: 影響を受けたコンポーネントを全て記載
   - 外部連携: 関わる外部サービスを全て記載
2. **削除・廃止の明記**: 何かを削除・廃止した場合、変更概要または該当セクションで必ず言及する。「追加」だけ書いて「削除」を漏らさない
3. **関連事項のセット記載**: 変更に関連する事項はセットで記載する（例: API エンドポイントを追加したなら、それを使うフロントエンド側の変更も記載する）
4. **反復チェック**: ドラフト完成後、各セクションを「このカテゴリで他に漏れている変更はないか」という視点で 2 周確認する

本ファイル内の従来記述と `github-pull-request` スキルの出力ルールが矛盾する場合は、スキルを優先する。

## 自動判定機能

パラメータが省略された場合の自動判定ロジック。

- **ベースブランチ**: `develop` → `main` → `master` の順で存在確認し、最初に見つかったブランチをベースとする
- **コミットタイトル**: ステージ済み変更を分析し、適切なプレフィックス（`feat` / `fix` / `docs` / `refactor` / `test` / `chore` 等）を判定
- **PR 説明**: `github-pull-request` スキルにより階層型分析に基づいて生成

## ワークフロー例

```bash
# 1. 変更をステージング
git add src/auth.ts src/login.tsx

# 2. pr-stage 実行（コミットメッセージは自動生成）
/pr-stage

# または、コミットメッセージを明示的に指定
/pr-stage "feat: ログイン機能を実装"
```

## コミットメッセージフォーマット

以下の Conventional Commits スタイルを使用する。

- `feat:` - 新機能の追加
- `fix:` - バグ修正
- `docs:` - ドキュメントのみの変更
- `style:` - コードの動作に影響しない変更（フォーマット等）
- `refactor:` - リファクタリング
- `perf:` - パフォーマンス改善
- `test:` - テストの追加・修正
- `chore:` - ビルドプロセスやツールの変更

**"comprehensive" なコミットメッセージを書く** - 1 行タイトルだけでなく本文（body）を含める。ゴールデンサークル理論（Why → How → What）は思考の指針として活用するが、形式やテンプレートとして固定しない。シニアエンジニアが書く洗練されたコミットとして、状況に応じて最適な構成・表現を選ぶこと。
